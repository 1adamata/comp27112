01 / THE PROBLEM

Why textures?

So far we've treated the appearance of a model as magic. We know how surfaces and lighting work — but what about the colour and detail of an object?

A model with no texture is dull. We can tell it's a cow because the geometry is good, but a default flat colour tells us nothing about the material. If we wanted a wooden floor we'd just get a whitish blob. What we really want is detail — a surface that looks like it's made of something.

The whole question of this chapter: how do we go from the blank model to the textured one? The answer is textures.

02 / DEFINITIONS

Three ways to define a texture

“Texture” can mean several things. They split into three families.

▢ RASTER IMAGES

Image-based

Ordinary images mapped onto a surface — 1D, 2D, or even multi-dimensional. The 2D raster image is what graphics people usually mean by "texture", and it's the main focus here.

∿ PROCEDURAL

Procedural

Patterns generated by code — e.g. fractal functions in a fragment shader. No image file needed; the appearance is computed mathematically.

⛁ ADVANCED

Advanced structures

Richer data structures used mostly for offline rendering (animated films). Touched on at the end (see Ptex).

Procedural textures

Procedural textures are powerful: a small piece of fragment-shader code using fractal functions can produce intricate patterns. The same code can morph from a lava look, to scorched earth, to grass-with-snow, simply by changing colours and parameters. Behind the rich pattern is just maths telling the fragment shader how to colour each pixel, then slapping the result onto geometry.

FocusFrom here on, "texture" means a 2D raster image — the bread and butter of real-time graphics.
03 / THE MAPPING

Texels & the mapping problem

The job: take a point on the object, find the matching point on the texture, and copy the colour across.

We zoom into the texture to find the corresponding pixel, then map a pixel on the teapot to a pixel on the texture. A pixel of a texture has its own name — a texel (texture element). The name is self-explanatory.

Sounds simple — but the teapot is ultimately made of triangles. So really we map a triangle on the object to a triangle in the texture. This is exactly why we love rendering with triangles: a triangle is always flat, so flattening its coordinates into the texture's space is well-defined. To do this we need coordinates for the texture space — the notorious UV coordinates you've met in any modelling tool.

04 / IN THE PIPELINE

UV coordinates through the pipeline

The teapot is made of vertices. After the vertex shader everything is interpolated for rasterisation — so how does the texture coordinate survive the trip?

If we just dumped the texture into the fragment shader, defining the triangle's mapping there would be awkward. Instead we declare varying texture coordinates in the vertex shader. These are the UV coordinates defined at each vertex; they get interpolated through the pipeline so the fragment shader receives the correct per-fragment coordinate.

It's the same trick we already use for colour. Given a UV at each vertex of a triangle, we find the value at any interior point using barycentric coordinates:

Definition · UVUV coordinates are normalised 2D coordinates (0→1) stored per vertex that define how a texture image maps onto a 3D surface. They are interpolated across triangles and used in the fragment shader to look up the correct texel.
05 / UNWRAPPING

UV unwrapping & normalised coordinates

For a complex texture we must figure out how each surface point maps to the flat texture. That process is UV unwrapping.

One problem: what if the model suddenly becomes huge? A point measured in pixels stays fixed because raster textures are resolution-dependent. The fix is a second set of normalised coordinates — usually called s, t — that run 0→1 regardless of texture size. You supply a normalised texture and the GPU figures out the mapping to whatever resolution it needs.

06 / SEAMS & PROJECTION

Seams & UV projection

Achieving the mapping is, honestly, mostly manual labour. And it brings artefacts.

If a texture is too small for the surface, it repeats, and you see lines where the tiling meets. These repeat boundaries are called seams. We usually create them by flattening the model onto simpler geometry (e.g. cylinders split into quads), applying the texture there, then wrapping it back. On a complex model like the cow, badly-placed seams show as visible patch artefacts — meaning the texture is ill-defined or the seams are wrong.

Projection & implicit mappings

Simple shapes have implicit mappings. A cuboid always gets a cross-shaped projection; a banana is essentially cylindrical. Complex characters have many seams, so there's a higher chance of things going wrong. You'll also often see one texture file packed with lots of information and reused across objects to conserve memory.

Reality checkEven with machine-learning tools, UV mapping still needs heavy manual work — an artist fights the unwrap, flattens, re-adjusts UVs, repeats.
07 / THE CATCH

Resolution mismatch

If the texture resolution exactly matched the object on screen, life would be easy. It almost never does.

With 4K displays, a texture near the camera is often much smaller than the screen area it covers — you can't ship terabytes of texture per game. Other times the texture is much larger than the tiny region it occupies. Both cause artefacts, and each needs a different fix.

↑ MAGNIFICATION

Texture < Object

Texture is stretched over a large area → blocky pixelation. Fixed by filtering / sampling.

↓ MINIFICATION

Texture > Object

Many texels squeezed into few pixels → aliasing & shimmering. Fixed by mipmapping (and anisotropic filtering).

08 / TEXTURE < OBJECT

Texture filtering & sampling

When the texture is too small, the GPU stretches it and we get an annoying pixelated look. We want a smooth result instead.

Key ideaImages are made of pixels, and pixels are samples — not little squares. Hence "sampling". How we reconstruct a colour between texels defines the filtering method.

The cost of smoothness

Smoother filters cost more texture fetches. Nearest is a single lookup (very fast). Bilinear needs 4 fetches + 3 linear interpolations. Bicubic needs 16 fetches + cubic polynomials — quite slow, though modern GPUs are tuned for it. The choice can even be aesthetic: a retro pixel-art game may want nearest.

Hard limitNo purely deterministic algorithm can beat the original raster resolution. Filtering reconstructs between texels; it cannot add information that was never captured.
09 / TEXTURE > OBJECT

Minification & aliasing

Textures are usually small — surely the texture is never bigger than the object? It happens extremely often.

Take a 512×512 texture on an object that ends up 10×10 pixels on screen — we'd represent ~100 screen pixels using 250,000+ texels. That's one of the most expensive things the GPU can do, and it's wildly unreasonable.

Worse: when a high-res texture recedes into the distance, many texels map to a single screen pixel. Bilinear only samples the 4 texels around your UV, so the receding pixels — whose true footprint is a big, stretched, irregular shape — miss most of the texels that should contribute. You get nasty aliasing, and it shimmers badly when moving (spatial-temporal aliasing).

Lab 3 · The receding checkerboard

A checkerboard floor in perspective. With naive point sampling, the distance dissolves into a writhing moiré pattern — that's aliasing. Toggle pre-filtering on and the far field correctly blurs toward grey (the average of black & white), just like a real photo.

10 / THE FIX

Mipmapping

Probably the most popular texture-filtering method ever — virtually every GPU does it.

“MIP” is from the Latin multum in parvo — “much in little.” The idea: pre-filter (downsample) the texture into a pyramid of ever-smaller levels. As an object recedes, the GPU picks the level whose texel size best matches the projected screen area — no more cramming 250k texels into 100 pixels.

Lab 4 · Mipmap level selector

Slide the camera away from the surface. The GPU swaps to a coarser pre-filtered level as distance grows. Turn on trilinear to blend smoothly between the two nearest levels instead of snapping.

Trilinear interpolation

Abruptly swapping levels shows obvious banding. So we take the two nearest levels, bilinearly sample both, then linearly interpolate between them — bilinear ×2 + one more lerp = trilinear interpolation. The mipmapped plane loses the wiggly moiré; on a black/white texture the far field averages to grey.

Assertion–reason (exam favourite)Mipmaps improve rendering performance — true, because they reduce cache misses by using appropriately-sized texture versions and reduce aliasing. The reason correctly explains the statement.
11 / GOING FURTHER

Anisotropic filtering

Mipmaps treat all directions equally. But a pixel's footprint is often stretched, not square.

Wrap a mipmapped texture on a cube and the front face looks fine, but the top face — far away and seen at a grazing angle — degrades into a grey, blurry trapezoid. Its footprint is a long, narrow shape, so isotropic mipmapping picks a very coarse level to fit it, throwing away detail along the un-compressed axis.

Anisotropic means “not the same in all directions” (from Greek). The filter acknowledges the elongated footprint: it samples more along the compressed axis and less along the others. A ground plane at a grazing angle is squashed in depth but barely across width, so the filter behaves differently per direction. Isotropic mipmapping reduces resolution equally in U and V — fine for a square footprint, wrong for a stretched one.

CostAnisotropic is much pricier than mipmapping — in the worst case 16× more texture fetches (the “Anisotropic ×16” setting). Turn it on and you may see frame drops. Use it sparingly.
12 / HARDWARE & API

Texture units & the code

Where do all these interpolations actually run? Not in the fragment shader — that'd be far too expensive.

The fragment shader is a general-purpose unit running across every pixel; filtering needs heavy caching and indexing. So GPUs have dedicated texture units — specialised hardware that is extremely fast at texture operations. Because of them we can render scenes with thousands of textures in real time. Software filtering in the fragment shader is possible but reserved for special cases.

In WebGL the texture unit is explicit. You create a texture, pick a texture unit (e.g. TEXTURE0), upload the image, set the filtering (min/mag) and wrap (s/t) parameters, generate mipmaps, then bind it to a sampler2D uniform. The varying UV from the vertex shader does the lookup.

const gl = canvas.getContext("webgl");
const texture = gl.createTexture();

gl.activeTexture(gl.TEXTURE0);          // choose the texture unit
gl.bindTexture(gl.TEXTURE_2D, texture);

gl.texImage2D(
  gl.TEXTURE_2D, 0, gl.RGBA,
  512, 512, 0,
  gl.RGBA, gl.UNSIGNED_BYTE, pixelData
);

// minification: trilinear (bilinear + between mip levels)
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MIN_FILTER, gl.LINEAR_MIPMAP_LINEAR);
// magnification: bilinear
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MAG_FILTER, gl.LINEAR);
// what to do at the seams — repeat in s and t
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_S, gl.REPEAT);
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_T, gl.REPEAT);

gl.generateMipmap(gl.TEXTURE_2D);

gl.useProgram(shaderProgram);
gl.uniform1i(gl.getUniformLocation(shaderProgram, "u_texture"), 0);
precision mediump float;

uniform sampler2D u_texture;   // the bound texture unit
varying vec2 v_uv;            // interpolated from the vertex shader

void main() {
  gl_FragColor = texture2D(u_texture, v_uv);
}
// three.js abstracts all of the above away
const texture = new THREE.TextureLoader().load("checkerboard.png");

texture.minFilter = THREE.LinearMipmapLinearFilter;
texture.magFilter = THREE.LinearFilter;
texture.wrapS = THREE.RepeatWrapping;
texture.wrapT = THREE.RepeatWrapping;

const material = new THREE.MeshBasicMaterial({ map: texture });
const mesh = new THREE.Mesh(geometry, material);
scene.add(mesh);
Translating the paramsLINEAR_MIPMAP_LINEAR = trilinear (bilinear within a level + linear between levels). MAG_FILTER = LINEAR = bilinear when magnifying. WRAP_S/T = REPEAT tiles the texture at the seams, using the normalised s,t coords.
13 / ADVANCED

Per-face textures (Ptex)

A different approach from Disney Animation — eliminate UV unwrapping and seams entirely.

Ptex stores texture data per face rather than requiring a UV layout. That kills seam artefacts and is excellent for subdivision surfaces and high-end film. The trade-offs: much larger files, poor real-time performance, limited software support, and clumsier artist workflows (you can't paint as easily as on a flat 2D UV map).

In practice, traditional UV-based raster textures remain the industry standard — universally supported, memory-efficient, real-time-friendly, and intuitive to paint. Ptex is reserved for offline film work where render quality justifies the limitations. UV textures dominate everywhere else: games, real-time apps, VR.

Zoom outThe rendering pipeline keeps growing more complex — but we've been chipping away at it. Texturing slots neatly into the pipeline you already know: vertices carry UVs, the rasteriser interpolates them, the texture unit filters the lookup, the fragment shader writes the colour.

15 / EXAM PRACTICE

Past-paper questions

Texturing recurs in COMP27112 exams — most often as bump mapping: making a surface look bumpy without changing its geometry. Expand each for a model answer keyed to this chapter.

2016Q2(d) — Describe an image-based technique to make a mesh M appear to have an irregular, bumpy surface, applied during rendering.4 marks
"You would like M to appear to have an irregular, bumpy surface. Describe an image-based technique which could be applied during rendering to achieve this."
Model answerUse bump / normal mapping: an extra image-based texture supplies a per-texel perturbation of the surface normal. During shading, the interpolated normal is tilted by the values read from this map before the lighting equation (diffuse/specular) is evaluated per fragment. Because only the normals change — not the geometry — the lit surface gains the illusion of bumps and dents at no extra polygon cost. (A normal map stores the perturbed normals directly, typically appearing blue-purple because most normals point out of the surface along +z.)
2017Q11 — Describe a method for making a surface look bumpy that does not change the actual geometry.2 marks
"Describe a method for making a surface look bumpy that does not involve changing the actual geometry of the surface."
Model answerBump mapping. Store a texture of normal perturbations; during fragment shading, perturb the surface normal per texel and feed the modified normal into the lighting calculation. The geometry stays a smooth low-poly mesh, but the per-pixel shading varies as if the surface were bumpy.
2021/22Q — Bump mapping: meaning of bvNv and buNu; run-time vs precomputed.essay
"The process of bump mapping is explained by the following diagram. What do the terms bvNv, buNu mean? How do they contribute to creating a surface that looks bumpy? Is it better to compute these values at run time or in advance? Give examples where live and precomputed bump maps would be used."
Model answerThe bump map is a height field b(u,v). Its partial derivatives bu and bv (rate of change of height along the u and v texture axes) scale the surface tangent vectors Nu, Nv; adding buNu + bvNv to the original normal tilts the normal to mimic the slope of the bumps. The lighting then responds to these perturbed normals, producing convincing highlights and shadows of a bumpy surface — without altering geometry. Precompute (bake) the perturbed normals into a normal map for static surfaces where the bump pattern is fixed (most game assets) — it's cheaper at run time. Compute live when bumps are dynamic or procedural (e.g. ripples on animated water, deforming surfaces) where baking can't capture the change.
2025Q — How can a surface be made to appear bumpy without changing its geometry?—
"How can a surface be made to appear bumpy without changing geometry?"
Model answerBump / normal mapping — perturb the surface normal per fragment using values sampled from a texture, then light using the perturbed normal. Contrast with displacement mapping, which does move real vertex positions (and so needs a finely-tessellated mesh to show any effect — a coarse mesh has too few vertices to displace).
Why these belong hereBump mapping is an image-based texturing technique — the same UV-mapped raster texture machinery from this chapter, but the image stores normals/heights instead of colour. Filtering, mipmapping and wrap modes all still apply.
