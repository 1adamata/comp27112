01 · orientation

From your hands to the display

Before the camera "sees" anything, the GPU quietly does a huge amount of work. The full system is a loop: input devices feed the graphics API, which talks to the application program and its model; the model is rendered and the API decides how to draw it back to the display.

The honest disclaimer. The lecturer's giant "unified" research pipeline (below) is not examinable — it's there to show how much real graphics systems abstract away. What you do need is the small simplified pipeline we build up in section 05.
02 · graphics APIs

The software that talks to your GPU

To reach pipeline steps and command the GPU/driver we need a graphics API — an intermediate layer. Tap a card to expand.

Key facts to keep. OpenGL = released 1992, portable, the ES suffix means Embedded Systems. WebGL evolved from OpenGL, runs in the browser via JavaScript, compiled at runtime. three.js is a wrapper on top of WebGL. OpenGL, WebGL & Vulkan are maintained by the Khronos Group (cross-platform). Vulkan = strongest control but ~800 lines just to draw a cube. DirectX = Microsoft, Metal = Apple.
03 · the hardware

CPU vs GPU — why graphics is parallel

A GPU is a many-core processor optimised for parallel work. That fits graphics perfectly: the same transformation runs on thousands of vertices, and the same shading runs on millions of pixels — all at once.

CPU

A few powerful cores. Runs the application and the graphics API (e.g. JavaScript + WebGL). Reads scene data from memory (RAM), decides what to draw, and sends the relevant data to GPU memory.

serial-strongorchestrates

GPU

Massively parallel cores + its own GPU memory. Takes the scene data and runs the pipeline operations on it, spitting out a finished image.

04 · scene data

What we actually feed in: vertices

The data is mostly scene parameters describing vertices — the building blocks of rendering. Connect vertices into polygons (almost always triangles). Think of a vertex as a container of attributes. Click a vertex to inspect it.

Vertex 0 · attributes

position (X,Y,Z)0.0, 0.0, 0.0
colour (R,G,B,A)0.91, 0.45, 0.23, 1.0
UV (textures)0.0, 0.0
timet
A vertex can carry positions, colours, UV coordinates (for textures) and even time — you decide based on your application. Positions tell the API "draw a triangle here".
05 · the pipeline · ⭐ examinable

Fixed vs Programmable

Older APIs used a fixed pipeline (rigid built-in steps). Modern graphics uses the programmable pipeline, where two stages are replaced by shader programs you write. Toggle below, then click any stage.

⭐ Exam-critical. Memorise the programmable order: 3D Vertices → Vertex Shader → Geometry / Tessellation → Rasterisation → Fragment (Pixel) Shader → Per-Sample Ops → pixel colours. The vertex shader & fragment shader are the two you must supply/write. The other stages typically run fixed on hardware.

Why choose Fixed?

Easier for beginners. But it's generally deprecated, slower, and not as well maintained on modern APIs.

Why choose Programmable?

Steeper learning curve, but full control and the ability to optimise for your exact app — e.g. an 8-bit indie game doesn't want costly ray tracing burning the GPU.

06 · drawing a triangle

Raw WebGL vs three.js

Same triangle, two levels of abstraction. three.js is a wrapper — the short version compiles down to roughly the long one. (Example code; not examined, but it shows the moving parts.)

// First link and compile Shaders
const program = gl.createProgram();
gl.attachShader(program, vertexShader)
gl.attachShader(program, fragmentShader)

const posLoc = gl.getAttribLocation(program, 'pos');
const vertices = new Float32Array(
[ -1, -1, 0, // vertex 1
   1, -1, 0, // vertex 2
   0,  1, 0  // vertex 3
]);
vertex_buffer = gl.createBuffer();
gl.bindBuffer(gl.ARRAY_BUFFER, vertexBuffer);
gl.bufferData(gl.ARRAY_BUFFER, vertices, gl.STATIC_DRAW);
gl.vertexAttribPointer(posLoc, 3, gl.FLOAT, false, 0, 0);
gl.enableVertexAttribArray(posLoc);
gl.drawArrays(gl.TRIANGLES, 0, 3);
const geometry = new THREE.BufferGeometry();

const vertices = new Float32Array(
[ -1, -1, 0, // vertex 1
   1, -1, 0, // vertex 2
   0,  1, 0  // vertex 3
]);
geometry.setAttribute('position',
   new THREE.BufferAttribute(vertices, 3));

const mesh = new THREE.Mesh(geometry, shaders);
// "material" in tutorials == your shaders
What's happening. Create a buffer in GPU memory, bind it (tell the GPU "use this one"), fill it with vertices (a flat 1-D array), then describe how to read it — 3 components per vertex (X,Y,Z), as floats. Finally drawArrays(TRIANGLES, 0, 3) draws from vertex 0 to 3. In WebGL you must state the primitive (TRIANGLES); a three.js Mesh defaults to triangles.
07 · shaders in GLSL

The two programs you write

WebGL/OpenGL use GLSL (OpenGL Shading Language); DirectX uses HLSL — nearly identical, so learn one. The vertex shader processes each vertex independently and outputs gl_Position. The fragment (pixel) shader processes each pixel independently and outputs gl_FragColor.

Vertex Shader

Runs per vertex. Positions geometry in the scene.

→ sends out gl_Position

Fragment Shader

Runs per pixel/fragment. Computes lighting & texture colour.

→ sends out gl_FragColor

The three GLSL data qualifiers — tap each:

attribute
Per-vertex input. Read-only, only on the vertex shader (e.g. position).
uniform
Shared value accessible by both shaders (e.g. matrices, time).
varying
Passes data vertex → fragment; interpolated over screen space during rasterisation.
vertex shader · GLSL

The classic vertex shader. Note vec4 — GLSL promotes to homogeneous coordinates for the transforms. Order is right→left, so model-view applies before projection.

attribute vec4 gl_Vertex;       // incoming vertex from the buffer (read-only)
uniform gl_ProjectionMatrix;   // supplied via the API
uniform gl_ModelViewMatrix;    // we know how to build this

void main(){
  gl_Position = gl_ProjectionMatrix * gl_ModelViewMatrix * gl_Vertex;
  // gl_Position is passed down the pipeline
}
fragment shader · GLSL

Must output a colour. vec4 = (R, G, B, A) where A is the alpha/opacity. A varying is interpolated across the surface during rasterisation — combine it with a uniform time for animated effects.

// simplest: every pixel white & opaque
void main(){ gl_FragColor = vec4(1.0, 1.0, 1.0, 1.0); }

// animated: varying UV * uniform time on the red channel
varying vec2 uv_coord;
uniform float uTime;
void main(){
  vec3 color = vec3(uv_coord.x * uTime, 0.3, 0.1);
  gl_FragColor = vec4(color, 1.0);
}
08 · live shader playground

Run the fragment shader yourself

This is a real WebGL canvas running the fragment shader from the slides. The varying UV is interpolated across the surface; drag uTime to see the "spooky red effects" the lecturer mentions. Move your mouse over the canvas too.

What you're seeing: the red channel = fract(uv.x · uTime), green is the interpolated uv.y, blue is fixed. Because uv_coord is a varying, the rasteriser interpolates it smoothly between the triangle's corners — exactly the behaviour described in the lecture. 
precision mediump float;
varying vec2 uv_coord;
uniform float uTime;
void main(){
  vec3 c = vec3(fract(uv_coord.x*uTime),
               uv_coord.y*0.6+0.1, 0.25);
  gl_FragColor = vec4(c, 1.0);
}

11 · one-screen recap

Everything, distilled

APIs

OpenGL (1992, ES=embedded), WebGL (browser/JS, from OpenGL), three.js (wraps WebGL), Vulkan (max control). Khronos maintains all three; DirectX=MS, Metal=Apple.

Hardware

CPU runs app+API, GPU is massively parallel. Output lands in the framebuffer before going to the display.

Scene data

Vertices = containers of attributes: position, colour, UV, time. Connected into triangles.

⭐ Pipeline

Vertices → Vertex Shader → Geometry/Tessellation → Rasterisation → Fragment Shader → Per-Sample Ops → pixels.

Fixed vs Programmable

Fixed = easy but deprecated/slower. Programmable = steeper but full control; you write the vertex & fragment shaders.

GLSL qualifiers

attribute (per-vertex, read-only), uniform (both shaders), varying (vertex→fragment, interpolated).
