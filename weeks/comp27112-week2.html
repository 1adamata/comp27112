What is an image — and where do they come from?

An image is any 2-D grid of measurements: each sample sits at a location (x, y) and records some physical property. So "image processing" is far wider than photo filters — it's anything that produces a spatially coherent measurement.

Image ≠ photograph

A sample can measure intensity, temperature, X-ray absorption, reflectance or distance. A 3-D CT volume is as much an "image" as a holiday snap — IP is often about measurement & reconstruction, not just enhancement.

🧠 Memory hook — "measure, not photograph"

Tim's CT-shoulder demo in Mango: many X-ray projections → reconstructed 3-D volume → browse slices or render surfaces. The "image" is the measurement.

Energy sources you can image with

Electromagnetic: gamma · X-ray · visible · infrared · microwave · radio.  Other: sound (ultrasound/sonar) · magnetic fields.

γ-rayX-ray / CTUVvisibleIRradarlidar

Different physics → different information: X-ray = absorption through the body; gamma = emission from a radiopharmaceutical; IR makes water look dark (easy rivers/coasts); lidar gives time-of-flight depth.

A little history — the three the exam likes

Bartlane transmitted photos over submarine telegraph cables with only ~5 grey levels — crippled by limited intensity resolution. Ranger 7 used computers to clean noisy moon images. CT made computation essential, not optional.

From the continuous world to a digital image

The real world is continuous in space and intensity. Making a digital image needs two separate discretisations — and confusing them is a classic exam trap.

Sampling — discretise space

Measure the image on a grid of pixels instead of continuous (x, y). 256×256 → 65,536 samples; 1920×1080 → ~2 million. Spatial resolution = how finely you sample space → controls detail / sharpness.

Quantisation — discretise intensity

Map continuous brightness to discrete levels. 1 bit → 2 levels; 8 bits → 256 greys; 12 bits → 4096 (common in medical). Intensity resolution = how many distinct values → controls smoothness / contrast.

ConceptDiscretisesControlsToo little →
SamplingSpace (x, y)Sharpness / detailblocky, blurry, aliasing
QuantisationIntensitySmoothness / contrastbanding / posterisation
Exam framing — don't mix these up

Increasing resolution does not fix poor quantisation, and more bit depth does not fix low spatial resolution. They're independent knobs. A digital image is always an approximation of reality.

Image as a function

f(x, y) → intensity // continuous f[x, y] ∈ {0, 1, …, L−1} // digitised · L = levels, e.g. 256

A colour image is just several greyscale images stacked (usually R, G, B), each with its own sampling + quantisation. Colour models came in Week 1 — this is intuition only.

3

Resolution — spatial, angular & ground

Three flavours. The two calculators show every step — there's no calculator in the exam, so the method is what scores.

Angular resolution — degrees per pixel

Each pixel sees a small slice of the field of view (FOV).

resolution = FOV ÷ number of pixels

Ground resolution — metres per pixel (aerial)

ground width = 2 · tan(FOV/2) · altitude ground res/px = ground width ÷ pixels

Workflow: (1) compute ground width, (2) divide by pixel width.

Intensity resolution

b bits → 2b levels. 8-bit grey = 256; 24-bit colour = 16.8M. Humans distinguish only ~40 shades of brightness and ~7.5M colours — so 8-bit often looks fine while quietly throwing away levels. Too few → visible banding / posterisation.

Nyquist (intuition only)

Sample at ≥ 2 × the highest frequency present, or information is lost forever and aliasing appears (jagged edges, moiré, wheels spinning backwards).

one-liner

To detect an object you need ~2 pixels across its smallest feature; to recognise it, more. Aliasing is a sampling problem, not noise.

Noise & signal-to-noise ratio

Noise is random variation in pixel values. It worsens in low light because the sensor amplifies a weak signal — and the noise with it.

SNR calculator (dB)

SNR = 20 · log₁₀( range ÷ σ )

Lecture: intensity 128, Gaussian noise mean 0, σ = 8 → ≈ 30 dB. Higher dB = cleaner. Logarithmic scale.

5

Point processing & the image histogram

Point processing means each output pixel depends only on the corresponding input pixel — never on neighbours (that's region processing, the convolution widget).

g(x,y) = T( f(x,y) ) // T = brightness · contrast · threshold · negative · solarise

What a histogram is

x-axis = grey level (0–255); y-axis = number of pixels at that value. Its shape tells you globally whether the image is dark, bright, or well-spread.

int histogram[256] = {0};
for (each pixel p in image)
    histogram[p]++;            // count each grey value

Read the shape

  • Dark image → mass piled near 0.
  • Bright image → mass piled near 255.
  • Good contrast → spread across many levels.
  • Comb / gaps → only a few distinct levels present (the "64 grey levels" slide looks fine; 32 or 16 starts to band in the sky).

Live histogram — built with the VTK kit

Pick a synthetic image and watch its 256-bin histogram. Bimodal images (Cells) have the two humps that make thresholding easy.

Brightness, contrast & clipping

Two of the simplest mappings — and the bug everyone hits if they forget to clamp the result.

Transfer-function playground

Pick an op; watch the graph of T (every input 0–255 mapped to an output) and a real intensity ramp transformed by it. Understanding the graph = understanding the op.

1.5

original ramp 0→255:

after T:

The maps

brightness: g' = g + k // shift, shape kept contrast: g' = g × k // stretch/compress

Brightness = shift. Contrast = scale.

Clipping (very important)

Results must stay in 0–255. With unsigned bytes, 255 + 1 = 0 — it wraps, so a too-bright pixel turns black. Compute in a wider type, then clamp:

uint8 px = 250;
int32 p = px;
p += 50;                 // increase brightness
if (p > 255) p = 255;
if (p < 0)   p = 0;
px = p;
Exam framing

Adding 50 doesn't only black-out the brightest pixels — values 206→255 wrap to 0→49 unless you clip. State it: g' = min(255, max(0, g + k)). Forgetting clipping is the single most common point-processing error.

7

Input–output mapping, LUTs & solarise

Stop thinking "image operation" and think "every input grey level g maps to an output g'." That single curve is the operation.

Two ways to implement a map

  • Formula per pixel: img[r][c] = img[r][c] * 1.5
  • Lookup table: precompute map[256], then img[r][c] = map[img[r][c]]

LUTs win — constant-time per pixel, and encode any non-linear curve consistently.

map[256] = [0, 2, 3, 5, 6, 8, 9, 11, …] g' = map[g]

Linear vs non-linear

A straight line = brightness/contrast. A curve (S-shape, tent) = non-linear. Solarise is the classic piecewise one:

g' = 255 − g if g > T g' = g otherwise

Values above T are inverted; below stay put → the "tent" transfer function.

Solarise tent & min–max stretch ramp (redrawn from the slides)

min–max: g' = 0 if g < min g' = 255 if g > max g' = (g − min)/(max − min) × 255 otherwise

Use it when intensities only occupy part of the range (e.g. 50–225) and the image looks flat — stretch the used band to fill 0–255. (Try it live in the OpenCV lab as normalize(..., NORM_MINMAX).)

8

Thresholding & automatic T

Thresholding turns an image binary to separate object from background. The whole game is choosing T.

g' = 255 if g ≥ T g' = 0 otherwise — output has just 2 levels, whatever T is

Manual

If the histogram is bimodal, pick T in the valley between the two humps. Needs a human → not scalable.

P-tile lecture

Object is a known P% of the image: expected object pixels = P% × total; accumulate the histogram until you reach that count; current grey level = T. Stable under lighting changes (uses proportions). Lecture: biscuit ≈ 41%.

Otsu OpenCV

Automatic. Picks T to maximise between-class variance (≡ minimise within-class). Best on a roughly bimodal histogram. Built into OpenCV.

Mean-splitting / midpoint — the simple automatic idea (≡ your Week-1 isodata)

for each candidate i: μl = mean intensity below i μh = mean intensity above/including i midpoint = (μl + μh) / 2 pick T = i where midpoint is closest to i

The interactive "auto-pick T" in the histogram & threshold widget animates exactly this convergence.

Otsu in OpenCV C++ marked

// supplied threshold
threshold(inputImage, outputImage, 128, 255, THRESH_BINARY);

// automatic Otsu threshold
threshold(inputImage, outputImage, 0, 255,
          THRESH_BINARY | THRESH_OTSU);

Try both live in the OpenCV lab below — Otsu reports the T it picked.

Misclassification (the limitation)

When object and background histograms overlap, any single global T misclassifies some pixels both ways — unavoidable with one threshold.

Geometrical transformations

Point processing changes pixel values; geometry changes pixel positions: (x, y) → (x', y'). Read every matrix as [new] = [matrix][old] — "row across × column down."

scaling (a,e) shearing (b,d) rotation (a,b,d,e) translation (c,f)

Scaling

x'y' = sx00sy xy

sx=sy=2: (1,0)→(2,0), (2,0)→(4,0). Holes appear → see interpolation.

Translation

x'y' = xy + txty

tx=160, ty=60: (1,0)→(161,60). Translation needs an addition — which is why we move to homogeneous coords ↓

Homogeneous coordinates — one matrix for everything

Add a 1 and pad to 3×3 so scaling+translation become a single matrix multiply (and matrices can be chained):

x'y'1 = sx0tx0syty001 xy1
Exam framing

Matrix multiplication is not commutative → order matters. "Scale then translate" multiplies the matrices in reverse order of application.

Reflection (flip y)

1000−10001

Usually translate centre→origin, transform, translate back (height 480 → ty=480).

Shear (λx=0.8)

1λx0λy10001

→ x' = x + 0.8y, y' = y. The off-diagonal terms do the shearing.

Rotation — drag θ and watch the matrix fill

45°
cosθ−sinθ0 sinθcosθ0 001

For θ=45°: cosθ = sinθ = √2/2 ≈ 0.707. To rotate about the image centre, combine translate→rotate→translate-back into one matrix.

Affine transform — the one big view, plus OpenCV

x'y'1 = abcdef001 xy1

a,e → scale · b,d → shear · a,b,d,e → rotation · c,f → translation.

Exam framing — which OpenCV function?

warpAffine() takes a 2×3 matrix (affine); warpPerspective() takes a 3×3 matrix (projective/perspective). Try warpAffine live in the lab.

Camera lens distortion & calibration

Real lenses bend straight lines. Two named shapes, two correction models, one OpenCV fix.

Barrel vs pincushion (redrawn)

No distortion
Barrel (bulges out)
Pincushion (pinches in)

Radial distortion

Depends on distance r from centre, r = √(x²+y²):

x_corr = x(1 + k₁r² + k₂r⁴ + k₃r⁶ + …) y_corr = y(1 + k₁r² + k₂r⁴ + k₃r⁶ + …)

Tangential distortion

Caused by lens/sensor misalignment:

x_corr = x + 2p₁xy + p₂(r² + 2x²) y_corr = y + p₁(r² + 2y²) + 2p₂xy
Exam framing

OpenCV stores the distortion as [k₁ k₂ p₁ p₂ k₃], finds it from multiple chessboard photos (calibration), and corrects with undistort(). Radial depends on r; tangential on misalignment.

Interpolation & forward/reverse mapping

After a transform, pixel coordinates are usually non-integer, but pixels live on an integer grid. Interpolation fills the gaps; the mapping direction decides whether you get holes.

Interpolation methods

  • Nearest neighbour — copy the closest source pixel. Fastest, lowest quality, blocky/jagged.
  • Bilinear — weighted blend of 4 neighbours; smoother.
  • Bicubic — more neighbours; smoothest, most expensive.
Exam framing

Higher quality → more computation. No formulas needed at this stage.

Forward vs reverse

AspectForwardReverse
Iteratesource → destdest → source
HolesYesNo
Interpolationawkwardnatural
Usedrarelyalmost always

Forward makes holes — reverse fills every pixel

The practical loop (OpenCV, graphics pipelines)

  • Loop over destination pixels.
  • Apply the inverse transform to find where each came from.
  • Interpolate the source image there.
  • Assign the value → every destination pixel gets a value, no holes.
🧠 Memory hook — "ask backwards"

Reverse mapping asks "where did this pixel come from?"; interpolation answers when the answer isn't a whole pixel. The two are inseparable.

▼ reveal

3
Why must you clip after a brightness add, and what goes wrong if you don't?
▼ reveal
Unsigned-byte arithmetic wraps: 255+1=0. Without clipping, bright pixels (e.g. 206–255 after +50) wrap to 0–49 → bright areas turn black. Compute in a wider type then clamp to 0–255.


  
4
Describe the P-tile thresholding algorithm and one advantage over manual selection.
▼ reveal
Expected object pixels = P% × total; build the histogram; accumulate counts until you reach that number; the current grey level is T. Advantage: automatic and stable under lighting changes because it uses proportions, not absolute values.

What does Otsu's method optimise, and when does it work best?
▼ reveal
It chooses T to maximise between-class variance (≡ minimise within-class variance). Works best on a roughly bimodal histogram.

Write the homogeneous matrix that scales by (sx,sy) and translates by (tx,ty), and say why we use homogeneous coordinates.
▼ reveal
[[sx,0,tx],[0,sy,ty],[0,0,1]]. Homogeneous coords let translation become a matrix multiply, so transforms can be composed into one matrix — but order matters (multiplication isn't commutative).


  
2
Which OpenCV function for an affine transform, which for perspective, and what size matrix does each take?
▼ reveal
warpAffine() — 2×3 (affine). warpPerspective() — 3×3 (projective).


  
4
Why is reverse mapping preferred over forward mapping for rotation/scaling?
▼ reveal
Forward (source→dest) leaves holes because non-integer coords don't land on grid pixels. Reverse (dest→source) iterates over destination pixels, inverse-maps each to the source and interpolates — so every destination pixel gets a value, no holes, and interpolation fits naturally.

▼ reveal
Radial (barrel/pincushion) depends on distance r from centre; tangential on lens/sensor misalignment. Parameters [k₁ k₂ p₁ p₂ k₃] are found by chessboard calibration and applied with undistort().
